## CS 255 Portfolio Reflection (DriverPass)

### Briefly summarize the DriverPass project. Who was the client? What type of system did they want you to design?

The DriverPass project focused on designing a system for a client named DriverPass. The goal was to create a system that helps students prepare for their driving tests by giving them access to practice tests, scheduling driving lessons, and managing appointments with instructors. The project required both a business requirements document and a system design document so the client’s needs could be translated into a workable system design.

### What did you do particularly well?

One thing I did well was organizing the requirements and connecting them to the system design. I think I did a good job identifying the main users of the system and showing how they would interact with features like scheduling, account access, and lesson management. I also put effort into the UML diagrams so the workflow and system behavior were easier to understand.

### If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?

If I could revise one part, I would improve some of the UML diagrams by making certain decision points and flows even clearer. I would also tighten up a few sections of the system design document so the wording is more consistent across requirements, technical details, and diagrams. That would make the final design easier for a development team to follow without having to interpret anything.

### How did you interpret the user’s needs and implement them into your system design? Why is it so important to consider the user’s needs when designing?

I interpreted the user’s needs by focusing on what the client wanted the system to do for students, instructors, and staff, then turning those needs into features and workflows. For example, the need to schedule lessons and manage availability was reflected in the scheduling process and related diagrams. It is important to consider user needs because a system can be technically correct and still fail if it is hard to use or does not solve the actual problem the users have.

### How do you approach designing software? What techniques or strategies would you use in the future to analyze and design a system?

I approach software design by starting with the problem first, then identifying users, system goals, and key tasks before thinking about technical details. This course helped me see the value of breaking the system into smaller parts using requirements documents, process modeling, and UML diagrams. In the future, I would continue using step-by-step analysis, clearer documentation, and early diagramming to catch issues sooner and make the design easier to build and maintain.
